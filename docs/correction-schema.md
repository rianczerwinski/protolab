# Correction and Rule Schemas

## Correction

A correction records a single error: what the protocol produced, what was
correct, and why. Stored in `corrections.toml`.

### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | auto | Sequential ID (`corr_001`, `corr_002`, ...). Generated by protolab. |
| `subject` | string | yes | Case identifier — what was being analyzed when the error occurred. |
| `step` | string | yes | Which decision point in the protocol failed. Free-form but should be consistent across corrections for clustering to work. |
| `protocol_output` | string | yes | What the protocol (or the AI following it) actually concluded. |
| `correct_output` | string | yes | What the correct conclusion was. |
| `reasoning` | string | yes | Why the correct output is right — the evidence or logic the protocol missed, misweighted, or misinterpreted. |
| `rule` | string | no | Optional generalizable discriminator. Not every correction yields a rule — some are one-off errors. When a correction reveals a structural principle, state it here. |
| `metadata` | dict | no | Arbitrary key-value pairs. Adapter imports populate this with framework-specific data (scores, tokens, provider info). |

### Example

```toml
[[corrections]]
id = "corr_003"
subject = "evaluation_case_42"
step = "severity_classification"
protocol_output = "Classified as LOW severity based on single-occurrence criterion"
correct_output = "Should be MODERATE — the single-occurrence rule applies to isolated incidents, not to patterns with low frequency"
reasoning = "The protocol says 'single occurrence = LOW' but this case had three occurrences over 18 months. The rule's intent is about isolated vs. patterned behavior, not literal count thresholds."
rule = "When assessing severity, distinguish between truly isolated incidents and low-frequency patterns. A behavior occurring 2-3 times over an extended period is a pattern, not a single occurrence."
```

## Rule

A rule is a crystallized discriminator extracted from one or more corrections.
Rules are the intermediate representation between raw corrections and the
integrated protocol. Stored in `rules.toml`.

### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | auto | Sequential ID (`rule_001`, `rule_002`, ...). |
| `decision_point` | string | yes | Which step in the protocol this rule governs. |
| `rule` | string | yes | The generalizable discriminator — when you see X, it indicates Y rather than Z, because... |
| `confidence` | string | yes | How established this rule is (see below). |
| `source_correction` | string | yes | Which correction ID produced this rule. |
| `date` | string | auto | ISO timestamp of rule creation. |

### Confidence Levels

| Level | Meaning | When to use |
|-------|---------|-------------|
| `structural` | Follows from the domain's logic. Unlikely to be revised. | The rule is a necessary consequence of how the domain works, not an empirical observation. |
| `strong_pattern` | Observed multiple times. Could have edge cases. | Two or more independent corrections converge on the same discriminator. |
| `provisional` | Observed once. Plausible but unconfirmed. | A single correction suggests a general principle, but it hasn't been tested against other cases. |

Rules are automatically created at `provisional` confidence when a correction
includes a `rule` field. Confidence can be upgraded manually or by the
resynthesis process when multiple corrections support the same discriminator.

### Example

```toml
[[rules]]
id = "rule_002"
decision_point = "severity_classification"
rule = "Distinguish truly isolated incidents from low-frequency patterns. 2-3 occurrences over an extended period constitutes a pattern (MODERATE+), not a single occurrence (LOW)."
confidence = "provisional"
source_correction = "corr_003"
date = "2026-04-03T14:30:00"
```

## Preventable Errors

A correction is **preventable** if it targets a decision point that already has
an extracted rule — meaning the protocol (or the AI following it) should have
gotten this right based on existing learning. Preventable errors are tracked by
`protolab check` and serve as a resynthesis trigger: they signal that rules
exist but aren't being integrated effectively into the protocol.
